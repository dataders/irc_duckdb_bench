# DuckDB Iceberg REST Catalog Benchmark Report: `full-20260625T0915`

## Scope

- Targets: Lakekeeper local, Polaris local, Polaris remote, Snowflake Horizon.
- Run shape: locked/default variant, `tiny` and `small`, 3 repetitions each.
- DuckDB CLI: `/opt/homebrew/bin/duckdb` (`v1.5.4`).
- Workload per run: create table, insert generated rows, readback, delete even ids, read-after-delete verification, cleanup.
- HTTP capture: every run stores redacted stdout plus structured `http_debug_*.jsonl` extracted from DuckDB HTTP debug output after `CALL enable_logging('HTTP')`.

## Observations

- Fastest median workload wall time: Lakekeeper local at 0.089 s.
- Slowest median workload wall time: Snowflake Horizon at 22.476 s.
- Local Lakekeeper and local Polaris are sub-second for this `tiny`/`small` workload; local Polaris has somewhat more REST/table-create overhead than Lakekeeper.
- Remote Polaris is dominated by remote REST table commit/load plus object metadata/data calls; remote round trips make small-table operations much slower than local catalogs.
- Snowflake Horizon is dominated by REST table commit/load calls, especially create/insert/delete. The locked `stage_multi_metadata` variant passes, but Horizon has the highest end-to-end latency.
- `duckdb_logs` table output was captured as CSV for every run but contained no HTTP rows; the useful HTTP timing stream for this DuckDB CLI came through stdout after `CALL enable_logging('HTTP')`, now captured as structured JSONL.
